# Phase 3 提案書：UI / UX 改善

**作成日**: 2026-06-19
**対象期間**: Phase 1 安定後 〜 約2週間
**ゴール**: 「使いやすくて遅くないポータル」にする

---

## 📌 全体方針

Phase 1 が「動く」段階だったのに対し、Phase 3 は **「使い続けたくなる」段階** を目指す。
ファイルアップロード（Phase 2）より先に行う理由：

1. **ユーザー定着率が上がる** … 検索・フィルタなしではイベント数が増えた瞬間使いにくくなる
2. **Phase 2 のフィードバック収集に影響** … 使いにくいまま新機能を追加しても声が集まらない
3. **技術的に独立** … ファイル機能とUI改善は依存関係がない

## 🚀 すでに完了した改善（速度系）

| 改善 | 効果 |
|---|---|
| **localStorage キャッシュ** | 2回目以降は **約0秒で表示**。GASは裏で同期 |
| **祝日API 30日キャッシュ** | 起動時のネットワーク往復が1回減る |
| **FullCalendar 並列読み込み** | `defer` 属性で HTML パースをブロックしない |
| **同期ステータス表示** | ヘッダー右上に状態（🟢同期済み / 🟡キャッシュ表示中 / 🔴エラー）を表示 |
| **手動再読込** | 同期ステータスをクリックで強制リフレッシュ |

### 体感速度のビフォーアフター

```
【Before】 ログイン → 真っ白 → 3〜5秒待機 → ようやくカレンダー表示
【After】 ログイン → 即カレンダー表示（キャッシュ） → 裏で同期 → 静かに最新化
```

## 🎯 Phase 3 の改善メニュー（優先度順）

### 🥇 優先度A：必須機能（1週間）

#### A-1. イベント検索・フィルタ
**問題**: 直近5件しか見えない。過去/未来のイベントを探す手段がない。
**実装**:
- ヘッダー下に検索バー追加：キーワード（タイトル・場所・担当者）
- カテゴリチップ：全部 / 通常 / 学内 / 全体MTG / 幹部MTG
- 期間フィルタ：今後 / 過去 / 全て
**工数**: 0.5日

```
┌──────────────────────────────────────────────┐
│ 🔍 [検索...]  [全部] [通常] [学内] [MTG]  [今後 ▾]│
└──────────────────────────────────────────────┘
```

#### A-2. 直近イベントリストの正しいソート
**問題**: 現在は配列順なので過去イベントが上に来る。
**実装**: `Date >= today` でフィルタ ＋ 日付昇順ソート
**工数**: 30分

#### A-3. 担当者マスタ（メンバーリスト）
**問題**: 「鈴木」「すずき」など表記揺れが発生する。
**実装**:
- スプレッドシートに `Members` シート追加：名前・役職・有効フラグ
- 担当者入力欄をテキスト→ドロップダウン（オートコンプリート）に変更
- GAS API に `?action=members` を追加
**工数**: 1日

#### A-4. 削除UNDO
**問題**: 誤クリックでイベント消失。確認ダイアログだけでは不十分。
**実装**:
- 削除直後に画面下に「イベントを削除しました [元に戻す]」トースト
- 5秒以内のクリックで復元
- 内部的には GAS への削除リクエストを5秒遅延
**工数**: 0.5日

## 💡 設計上の注意点

### 状態管理
現在 `eventsData` というグローバル配列を全体で参照している。Phase 3 で検索・フィルタを足すと「絞り込み後の表示用配列」と「全データ」を分ける必要がある。

```js
let allEvents = [];      // 全データ（GASからの取得結果）
let filteredEvents = []; // 絞り込み後の表示用
let filterState = {      // 現在のフィルタ条件
    keyword: '',
    category: 'all',
    period: 'upcoming'
};
```

### キャッシュとデータ整合性
localStorage キャッシュがあるので、編集→保存後の挙動に注意：

- ✅ 保存成功 → eventsData を更新 → キャッシュも更新
- ❌ 保存失敗 → eventsData は更新しない → 警告表示

このルールを徹底すれば、リロード時に「保存したはずなのに古い」が起きない。

### モバイルでの編集体験
編集モーダルは現在大きすぎてスマホで操作しにくい。Phase 3 のモバイル対応で：
- フルスクリーンモーダル化
- フォームを縦並びに
- ボタンを画面下部固定

## 結論

Phase 3 では、**Phase 1 で完成した機能の「使い心地」を一気に底上げ**する。
特に **A-1（検索）** と **A-3（担当者マスタ）** はイベント数が増えると致命的な不便さを生むため、Phase 2 より先に行う価値が高い。

実装順は：
1. すでに完了済の **速度改善**（キャッシュ・defer・同期ステータス）
2. **優先度A**（4つ）：1週間
3. **優先度B**（7つ）：1週間
4. **優先度C** は任意・余裕があれば
