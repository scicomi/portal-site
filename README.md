カレンダー表示、イベント詳細管理、書類期限の自動計算、担当者管理などを行う。

## システム構成

```
┌───────────────────┐    ┌─────────────────┐    ┌──────────────────┐
│  GitHub Pages     │ ⇄ │  Google Apps    │ ⇄ │  Google          │
│  (フロントエンド)  │    │  Script (API)   │    │  Sheets (DB)     │
│  HTML/CSS/JS      │    │  Code.gs        │    │  SciComi_Portal_DB│
└───────────────────┘    └─────────────────┘    └──────────────────┘
        ↑                                                 ↑
   メンバーがアクセス                                  幹部が直接編集も可
```

| レイヤー | サービス | 役割 |
|---|---|---|
| **フロントエンド** | GitHub Pages | 静的HTML/CSS/JSを配信。無料・自動デプロイ |
| **バックエンドAPI** | Google Apps Script (GAS) | REST APIとして動作。CRUD操作・認証 |
| **データベース** | Google Sheets | イベント情報を保存。幹部が直接スプレッドシート編集も可能 |

### なぜこの構成か

- **無料**：サークル予算ゼロで運用可能
- **専門知識不要**：サーバー管理・データベース管理が一切不要
- **代替手段あり**：万一サイトが壊れてもスプレッドシートを直接見れば情報は失われない
- **既存ツールとの親和性**：サークルメンバーは全員Googleアカウントを持っている

## ページ構成

ポータルは4ページ構成：

| ページ | URL | 役割 |
|---|---|---|
| ホーム | `index.html` | ダッシュボード。直近イベント・期限・メンバー数等の要約 |
| イベント | `events.html` | カレンダー＋イベント管理（CRUD） |
| メンバー | `members.html` | 3カテゴリ（アドバイザー/コーディネーター/メンバー）テーブル表示 |
| 実験内容 | `experiments.html` | フィルタ式（工作/実験ショー/その他）の実験データベース |

```
├── events.html        # イベント管理
├── members.html       # メンバーリスト
├── experiments.html   # 実験内容
├── style.css          # 共通スタイル
├── config.js          # ★一元設定（URL・カテゴリ定義・期限ルール等）
├── api.js             # GAS通信レイヤー（3リソース対応）
├── app.js             # 共通ロジック（認証/ナビ/同期/トースト/ユーティリティ）
├── home.js            # ホームページ
├── script.js          # イベントページ
├── members.js         # メンバーページ
├── experiments.js     # 実験ページ
├── README.md
├── gas/
│   ├── Code.gs        # GASに貼り付けるバックエンド（3リソース対応）
│   ├── ImportData.gs  # 過去スケジュール一括取り込み（importSchedule）
│   ├── appsscript.json # GASプロジェクトマニフェスト（clasp用）
│   ├── .clasp.json    # clasp設定（scriptId要設定）
│   └── .claspignore   # clasp push 除外リスト
├── .github/
│   └── workflows/
│       └── deploy-gas.yml  # GitHub Actions: gas/ 変更時に自動 clasp push
└── docs/

    ├── 02_setup_guide.md
    └── 03_phase3_proposal.md
```

> **スクリプトの読み込み順**は全ページ共通で
> `config.js → api.js → app.js → 各ページ.js`。
> config.js が他の全てより先に読まれる前提なので、この順序は変えないこと。

### 拡張するときはまず config.js

カテゴリ追加・色変更・URL変更・期限ルール変更は、原則 **config.js だけ** 直せば全ページに反映される設計です。

| やりたいこと | config.js の該当箇所 |
|---|---|
| GASのURLを変えた | `API_URL` |
| イベント種類を増やす | `EVENT_CATEGORIES` に1行 |
| 実験タブを増やす | `EXPERIMENT_CATEGORIES` に1行 |
| メンバー区分を増やす | `MEMBER_CATEGORIES` に1行 |
| 書類期限の日数を変える | `DEADLINE_RULES` |
| 期限警告の色しきい値 | `DEADLINE_ALERT` |
| リマインダー通知日数 | `REMINDER.days` |

### 各ファイルの役割

| ファイル | 役割 | 編集頻度 |
|---|---|---|
| `index.html` 等HTML | 各ページの構造 | UI追加時 |
| `style.css` | 全ページ共通スタイル | デザイン変更時 |
| `app.js` | 認証・ヘッダー・同期・トースト（全ページが読み込む） | 共通機能追加時 |
| `api.js` | GASとの通信。3リソース（events/members/experiments）対応 | **API_URL更新時のみ** |
| `script.js` | イベント管理ロジック | イベント機能追加時 |
| `members.js` | メンバー管理 | メンバー機能追加時 |
| `experiments.js` | 実験管理 | 実験機能追加時 |
| `home.js` | ダッシュボード | ホーム表示変更時 |
| `gas/Code.gs` | バックエンドAPI | **GASエディタで管理（GitHubは履歴用）** |

## 開発・編集方法

### 1. ローカルで編集する

```bash
git clone https://github.com/scicomi/portal-site.git
cd portal-site
# お好みのエディタで編集（VSCode推奨）
```

### 2. ローカルで動作確認する

Webブラウザで `index.html` を直接開いてもCORSエラーで動かないので、簡易サーバーを立てる：

```bash
# Python 3 がインストールされている場合
python -m http.server 8000

# または Node.js
npx serve
```

ブラウザで `http://localhost:8000` を開く。

### 3. GitHubにプッシュ＝自動的に本番反映

```bash
git add .
git commit -m "変更内容の説明"
git push
```

数分後に https://scicomi.github.io/portal-site/ に自動反映される。

### 4. GAS（バックエンド）を編集する場合

1. [script.google.com](https://script.google.com) を開く
2. プロジェクト `SciComi_Portal_API` を開く
3. `Code.gs` を編集
4. 保存
5. **デプロイ → デプロイを管理 → 編集 → バージョン: 新しいバージョン → デプロイ**

**重要**：GASは「保存」だけでは公開されない。**必ず「新しいバージョン」でデプロイ**すること。

### 5. GASの自動デプロイ（clasp）

`gas/` 配下の変更を `main` ブランチに push すると、GitHub Actions が自動で `clasp push` を実行する。

**初回セットアップ**:

```bash
# 1. clasp をインストール
npm install -g @google/clasp

# 2. Google アカウントで認証（ブラウザが開く）
clasp login

# 3. ~/.clasprc.json の内容をコピー
cat ~/.clasprc.json

# 4. GitHub リポジトリ Settings > Secrets and variables > Actions で
#    シークレット名: CLASP_TOKEN / 値: 上でコピーした JSON をそのまま貼る

# 5. gas/.clasp.json の scriptId を実際の GAS プロジェクト ID に書き換え
#    GASエディタ URL: https://script.google.com/home/projects/【このID】/edit
```

手動コピペ運用も引き続き可能。clasp が未設定の間は今まで通り GAS エディタで直接編集する。

## 運用・保守

### パスワードを変更したい

1. スプレッドシート `SciComi_Portal_DB` を開く
2. `Config` シート → `password` の値を新しいものに変更
   - 管理（幹部）パスワードは `admin_password`。サイト右上「管理」→「管理者設定」からも変更可
3. メンバーに新パスワードを共有
4. **パスワードを変更すると、既存のログインセッションは即座に無効化される**（全員が次回アクセス時に再ログインを求められる）。漏洩時の締め出しにも使える

### ログインのしくみ（ロール自動判別）

ログイン画面は**1つのパスワード欄**だけ。入力された値で自動的にロールを判別する：

- `Config` の **`password`（一般）** を入力 → メンバーとしてログイン（閲覧・編集）
- `Config` の **`admin_password`（幹部）** を入力 → **自動的に管理者モード**になり、`パスワード一覧` ページへ遷移

> 幹部パスワードはメンバー権限も兼ねる（管理者は全機能を使えるため）。
> ログイン後でも、ヘッダー右上の「管理者」ボタンから途中で管理者へ昇格できる（従来どおり）。

### パスワード一覧ページ（管理者専用）

外部サービス（モノタロウ等）の **URL・ログインID・パスワード** を管理者だけが閲覧・追加・編集・削除できるページ。

- ナビの「パスワード一覧」は**管理者ログイン時のみ表示**される（メンバーには見えない）。
- 各項目はトグルで開閉。パスワードは伏字（••••）で、👁ボタンで表示／コピー可能。
- **閲覧・編集とも管理者トークン必須**（API 側でもガード。メンバーのデータ一括取得 `listAll` には含めない）。
- データは `SciComi_Portal_DB` の **`Passwords` シート**に保存される。
- ⚠ **セキュリティ注意**：第三者サービスのパスワードが**平文**で保存される。スプレッドシートの共有範囲を幹部のみに限定し、画面共有・スクショの取り扱いに注意すること。より厳密に運用したい場合は専用パスワードマネージャ（Bitwarden 等）の利用も検討。

### この更新を反映するための手順（重要）

フロント（HTML/JS）は GitHub Pages で自動反映されるが、**バックエンド（`gas/Code.gs`）は手動デプロイが必要**：

1. `gas/Code.gs` を GAS エディタに反映（clasp または手動コピペ）
2. GAS エディタで **`setupSpreadsheet()` を1回実行**
   - `Passwords` シートを新規作成
   - `Config` に `gemini_model` キーを追加
3. **デプロイ → デプロイを管理 → ✏️ → バージョン: 新しいバージョン → デプロイ**
4. 旧バージョンが残っている間も、フロントは旧 `auth` に自動フォールバックするのでログインは止まらない（ただし「パスワード一覧」とロール自動判別は再デプロイ後に有効化）。

### 過去スケジュール（23〜26年度）を一括取り込みする

`gas/ImportData.gs` に、23〜26年度の年間スケジュール.xlsm から抽出した
**過去イベント150件**と**実験17件（工作8・実験ショー9）**が埋め込まれている。

取り込み手順（1回だけ）:
1. GASエディタで先に `setupSpreadsheet()` を実行（新カラム `SeriesKey` / `Positives` / `Reflections` / `FiscalYear` 等を追加）
2. 続けて `importSchedule()` を実行
   - `Events` シートに過去イベントを追記（`Date`＋`Title` が重複する行はスキップ）
   - `Experiments` シートに実験を追記（`Name` が重複する行はスキップ）
3. 同名イベントは `SeriesKey` で自動的に紐づき、イベントページに「N回目／通算M回」が表示される

再取り込みしても重複しないので、何度実行しても安全。
分類は機械抽出のため、カテゴリ（イベント／その他／全体MTG／幹部MTG）はUI上で適宜修正可能。

### 過去のイベントを直接修正したい

スプレッドシートの `Events` シートを直接編集してOK。
ただし以下のフィールドは特殊なので注意：
- `PartsList`：JSON文字列。形式を壊さないこと
- `Files`：JSON文字列。同上
- `SeriesKey`：同名イベントを紐付けるキー（開催回数の集計に使う）。空欄なら `Title` で自動集計
- `ID`：絶対に変更しない

### イベントをCSVでエクスポートしたい

スプレッドシート `Events` シートで **ファイル → ダウンロード → カンマ区切り形式(.csv)**

### ストレージ容量を確認したい

スプレッドシートには事実上容量制限なし（500万セルまで）。
Google Drive（将来Phase 2でファイル保存に使う）の容量は drive.google.com で確認。

### Phase 4: リマインダーメール・年間レポート

**リマインダーメール（毎日自動）**

イベントの許可願・報告書期限が近づくと、担当者とアドバイザー/コーディネーターに自動メールが届く。

セットアップ（1回だけ）:
1. GASエディタで `installTriggers()` を実行
2. Config シートで `reminder_enabled` が `true` になっていることを確認
3. `reminder_days` で通知日数を調整（デフォルト: 7日前, 3日前, 1日前）
4. メンバーページでアドバイザー/コーディネーターのメールアドレスを登録

**年間レポート**

年度の活動レポート（イベント数・実験使用ランキング・メンバー統計）をHTML形式でメール送信。

- **自動送信**: `installTriggers()` 実行済みなら、**毎年4月1日に前年度分が自動送信**される（`Config` の `annual_report_enabled` が `true` の場合）。
- **手動送信**: GASエディタで `generateAnnualReport(2026)` を実行（引数なしなら現在年度を自動判定）。
- 送信先は Config シートの `report_recipients`（空欄ならアドバイザー/コーディネーター全員）。

### Phase 2: ファイルアップロード (Google Drive)

イベント編集画面から関連ファイル（PDF、画像、Excel等）を Google Drive にアップロードできる。

- アップロード上限: 10MB/ファイル（`Config` シートの `file_max_mb`。フロントの上限と必ず一致させる）
- 保存先: Google Drive の `SciComi_Portal_Files` フォルダ（自動作成）
- 共有設定: `Config` シートの `file_sharing` で制御
  - `domain`（既定・推奨）: 同じ Google Workspace ドメイン内でリンクを知る人のみ閲覧可。**児童名簿や許可願など個人情報を含むファイルを扱うため、可能な限りこちらを使う**
  - `anyone`: リンクを知る全員が閲覧可（個人 Gmail 運用などドメインが無い場合）
  - ※ 個人アカウントで `domain` を指定しても共有不可なら、自動的に `anyone` にフォールバックする
- 自動削除: 5年以上前のファイルを毎月1日に自動ゴミ箱移動
  - `Config` シートの `file_retention_years` で保持年数を変更可能
  - ゴミ箱からは30日間復元可能
  - 注意: 作成日基準で削除するため、毎年使う催しの古い添付もゴミ箱に入る。重要ファイルは別途保管を推奨

初回セットアップ: `installTriggers()` を再実行（Phase 4 で実行済みなら不要、ただしファイルクリーンアップのトリガー追加のため再実行推奨）

### バックアップ

`installTriggers()` 実行済みなら、**毎月1日にスプレッドシートが自動バックアップ**される。
- 保存先: Google Drive の `SciComi_Portal_Backups` フォルダ（自動作成）
- ファイル名: `SciComi_Portal_DB_backup_YYYY-MM-DD`
- 保持世代数: `Config` の `backup_keep_count`（既定6世代。超過分は古いものから自動でゴミ箱へ）
- 手動で取りたい場合: GASエディタで `backupSpreadsheet()` を実行、またはスプレッドシート → ファイル → コピーを作成

### 操作履歴を確認したい（監査ログ）

`AuditLog` シートに、作成・更新・削除・ファイル操作・設定変更・認証失敗などが時系列で記録される。
- 共通パスワード運用のため「誰が」までは特定できない（トークンのハッシュとロールのみ）
- `Config` の `audit_keep_days`（既定365日）を超えた古い行は月次で自動削除される

### 同時編集について

2人が同じイベント/メンバー/実験を同時に編集して保存すると、**後から保存した側が「他の人が編集しました」と表示され、最新内容が再読込される**（上書き事故を防ぐ楽観的競合検知）。再度編集して保存し直せばOK。

## トラブルシューティング

### ログインしても何も起こらない

ブラウザで F12 → Console タブを見る：

```js
// 現在のトークンを確認
localStorage.getItem('scicomi_portal_token')
```

→ `null` → ログインに失敗している。パスワードを再確認。
→ 何かの文字列 → トークンは保存済み。GAS側の問題の可能性。

### 「unauthorized」エラーが出る

```js
// localStorage の値とスプレッドシート Config の password が一致しているか確認
console.log(localStorage.getItem('scicomi_portal_token'));
// → スプレッドシートで Config > password の値と比較
// → 違ったら以下で初期化して再ログイン
localStorage.clear();
location.reload();
```

### 「Failed to fetch」エラー

- `api.js` の `API_URL` がGASのデプロイURLと一致しているか確認
- GASのデプロイ設定が「アクセスできるユーザー: 全員」になっているか確認

### 「CORS policy」エラー

GAS Web Appは正常な200レスポンス時のみCORSヘッダーを付ける。
**実体は GAS が 500 エラーを返している**ことが多い。

1. GASエディタを開く
2. 左メニュー「実行数」を開く
3. 直近の❌（失敗）の詳細を見る
4. エラーメッセージから原因を特定

### セキュリティ更新版（トークン署名方式）をデプロイした直後の注意

トークンの仕組みを「ステートレス署名方式」に変更したため、**更新版を新バージョンでデプロイすると、既存のログインは一度だけ無効化される**（全員が再ログインを求められる）。これは想定通りの挙動。あわせて以下を1回だけ実施：

1. GASエディタで `setupSpreadsheet()` を実行
   - 新しい Config キー（`file_max_mb` / `file_sharing` / `annual_report_enabled` / `backup_keep_count` / `audit_keep_days`）を追加
   - `AuditLog` シートに `Role` 列を追加（マイグレーション）
2. `Config` シートで `file_sharing` を確認（既定 `domain`。個人 Gmail 運用なら `anyone`）
3. `installTriggers()` を再実行（年間レポート・自動バックアップ・孤児ファイル整理・監査ログ間引きのトリガーを追加）
   - 初回はバックアップ用に Drive への追加権限承認を求められる場合がある

### Bot（Gemini）が「レート制限／無料枠の上限」になる

APIキーを設定しても Bot が AI 回答にならず「キーワード検索に切り替えます」となる場合、ほとんどは **Gemini 無料枠の上限（429）** が原因。キー自体が無効なら別メッセージ（`APIキーが無効です`）になるので、429 = **キーは有効だが回数制限に当たっている**。

**最有力の原因: モデルの廃止（2026年6月）**

`gemini-2.0-flash-lite` と `gemini-2.0-flash` は **2026-06-01 に廃止（deprecated）** された。廃止モデルは完全停止（404）ではなく **無料枠の割り当てがほぼ没収**されるため、**1回のリクエストでも即 429**（`generate_content_free_tier_requests` quota exceeded）になる。
→ **対処**: 現行の安定モデルへ切り替える。本更新で既定を **`gemini-2.5-flash-lite`** に変更済み。`Config` に古い `gemini_model` が残っていても、サーバー側が自動で現行モデルへ置き換える（`setupSpreadsheet()` 実行時にも移行）。「管理」→「使用モデル」からも変更可。現行候補: `gemini-2.5-flash-lite` / `gemini-3.1-flash-lite` / `gemini-2.5-flash`。

**なぜ起きるか（無料枠の特性）**

- 無料枠の上限は **APIキー単位ではなくプロジェクト単位**。`gemini-2.0-flash-lite` で **15 RPM（1分あたり15回）/ 1,500 RPD（1日あたり1,500回）** 程度（Googleが予告なく変更・地域差あり）。
- このサイトは**1本のサーバー側キーを全メンバーで共有**するため、同時に数人が Bot を使うとすぐ「1分あたり」の上限に達する。
- RPD（1日上限）は**太平洋時間0時（日本時間 約17時）にリセット**。

**今回の更新で改善した点**

- 429 の**実際の原因（1分枠か / 1日枠か）を判別**してメッセージを出し分け：
  - 「1分あたり」→ 自動で数十秒待って**1回だけ自動再試行**。それでもダメならキーワード検索へ。
  - 「1日あたり」→ 当日は回復しないので、リセット時刻を案内してキーワード検索へ。
- API未有効化・地域制限・請求設定の問題（403）は **Google からの本文をそのまま表示**して原因を特定できるようにした。
- **使用モデルを切り替え可能**にした（`Config` の `gemini_model`、または「管理」→「使用モデル」）。キーがそのモデルの無料枠を持たない／モデル廃止時に、コード変更なしで別モデルへ。

**それでも頻発する場合の対処**

1. 「管理」→「使用モデル」で別モデル（例: `gemini-2.5-flash-lite`）に切り替える。
2. Google AI Studio でキーの**残量・無料枠の有効/地域**を確認。無料枠が出ない地域・アカウントなら、請求先を有効化（従量課金）すると上限が大幅に上がる。
3. 利用集中時間帯を避ける／Bot の利用を控えめにする。
4. 恒久対策が不要なら、Bot は未設定でもキーワード検索で動く。

### GASのコード変更が反映されない

GASは保存だけでは公開されない。
**デプロイ → デプロイを管理 → ✏️ → バージョン: 新しいバージョン → デプロイ** が必要。

### スプレッドシートを編集したのにサイトに反映されない

サイトはロード時にしかGASから取得しない。**ブラウザをリロード**すれば反映される。
Phase 3 で自動リフレッシュ実装予定。

### 既知の制約・将来課題（未実装。実装には仕様判断が必要）

- **個人認証なし**：共通パスワード方式のため、監査ログで「誰が」までは特定できない。Google ログイン（GAS の `Session.getActiveUser()`）への移行で解決できるが、全員に閲覧権限付与が前提。
- **メンバーは1年度1行モデル**：複数年在籍や昇格（member→coordinator）の履歴は持てない。年度をまたぐ場合は各年度に行を作る運用。横断管理が必要になればスキーマ再設計を検討。
- **実験の資料URLは1つ（`SlidesURL`）**：複数資料（指導案・動画等）を持たせるには `Files` 同様の配列化が必要。
- **キャッシュのスキーマ二重持ち**：イベントはページにより UI形/GAS形でキャッシュされる（防御的パースで実害はないが、将来はサーバー正準形に一本化推奨）。
- **PWA化・iCalエクスポート・通知の多様化（Teams/LINE）・分析ダッシュボード**：いずれも独立した機能追加として将来検討。

### よくある質問

**Q. 開発者が引退した後にバグが出たらどうする？**

A. 主な選択肢：
- スプレッドシートは無傷なので、最悪サイトが壊れてもデータは失われない
- 知り合いのエンジニアに依頼する（このREADMEを渡せば1日で理解可能）
- 復旧諦めてスプレッドシート直接運用に戻る（ポータルの本質的な意義は失われるが、データは保持される）

**Q. サイトをまるごと別の場所に移したい**

A. このリポジトリをフォークして、`api.js` の `API_URL` を新しいGASに向ければOK。
HTML/CSS/JS は完全に静的なので、Netlify、Vercel、Cloudflare Pages など好きなホスティングに移せる。

**Q. データだけ取り出したい**

A. スプレッドシートを直接開いてCSVエクスポート。技術力ゼロでも可能。
